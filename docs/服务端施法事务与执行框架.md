# 服务端施法事务与执行框架

## 责任与边界

`com.mcnoita.wand.server.CastTransaction` 是服务端唯一允许改变法杖状态的施法协调器。纯 evaluator 只产生绑定单一 catalog snapshot 的冻结 `ResolvedCast`；事务层把玩家持有的 `ItemStack`、snapshot、中央预算 reservation 和类型化执行器连接起来。它不重新解释卡牌顺序，也不接受客户端提供的伤害、随机种子、payload、目标、实体类型或世界操作。

当前权威入口只接受玩家选中的主手热栏槽。玩家必须存活、非旁观、未移除，且没有打开非玩家 `ScreenHandler`；不满足这些条件的包在读取或写入法杖前拒绝。

正式状态机为：

```text
CastIntent
  -> Validate source stack and player
  -> Evaluate on an ItemStack copy against one CatalogSnapshot
  -> Preflight root Trigger capacity and freeze synchronous query envelopes
  -> Reserve one BudgetRequest with a pre-generated execution UUID
  -> Revalidate CastBinding
  -> Write a replacement ItemStack copy
  -> Revalidate, commit the reservation, and replace the held stack on the server thread
  -> Execute frozen EffectPlan
```

`CastBinding` 绑定 player UUID、hand、slot、item ID/count、StateRevision、spells hash、规范化 NBT SHA-256、catalog epoch/hash 和 sequence。重读中的任一字段不一致都会拒绝本次 cast。`PreparedCast` 只持有隔离副本、已接受的 `ResolvedCast`、同一 snapshot、execution UUID 和对应的 `BudgetRequest`；`CommittedCast` 才表示 matching reservation 已提交且原手持栈已被替换。

## 原子性与失败

解码、迁移、冷却检查、外部热栏候选读取和 evaluator 都只接触 `ItemStack.copy()`。只有写入已暂存 replacement、再次校验 binding、提交 reservation 后，事务才替换真实手持栈。因此验证、求值、预算预留、暂存写入或任一提交前 binding 校验失败时，原法杖的 mana、uses、Deck、cooldown、revision 和 NBT 不变。GUI 保存法术会递增 `StateRevision`；打开任意非玩家 `ScreenHandler` 时直接拒绝施法，避免编辑与 cast 竞争。

在第一次纯求值前，客户端 binding 已经通过验证；在预算预留前生成 execution UUID。reservation 采用 `reserved -> committed -> closed` 生命周期并且为全有或全无。提交后，执行器只消费冻结 plan；单节点失败被隔离、记录限频诊断并幂等释放未使用的节点额度，绝不回滚已提交的法杖。`CastResult` 区分验证、求值、预算、旧 binding 和提交拒绝；只有 `ACCEPTED` 结果带有 `CommittedCast`。

## 适配器与网络

`MinecraftWandAdapter.readReadOnly`、`canCastReadOnly` 和 `canonicalNbtStateHash` 是只读边界，HUD 和二次 binding 不得因为 legacy migration 修改真实栈。`MinecraftExternalSpellPoolAdapter` 同样只读取其他热栏法杖的副本。

`CastIntent` 容纳 hand、slot、sequence 和可选 `ClientCastBinding`。协议 v2 将客户端观察到的 revision、catalog epoch/hash 和 NBT state hash 映射到该 binding；`CastTransaction` 在求值前、预算预留后以及 replacement 暂存后都重新比较服务器观察到的 binding。`MinecraftWandAdapter.stateRevisionReadOnly` 同样不会迁移真实栈。客户端 HUD 同步只作为下一次请求的 optimistic binding，不能授权任何世界效果。

## 预算与执行

`CastTransaction` 在 reservation 前把中央每次施法的 `AUTHORITATIVE_ENTITIES` 和 `TRIGGER_RELEASES` 上限传给纯 `RootTriggerBudgetAllocator`。根投射物与其冻结 Trigger 子树无法获得足够本地 runtime budget 时，会带 `PER_CAST` 结构化诊断拒绝，真实法杖 NBT 不变；该 ceiling 也被写入实体运行时预算，最大硬上限为 128。`PlanBudgetRequestFactory` 从 `ResolvedCast` 的 action/projectile/payload usage 与 typed effect nodes 生成 `BudgetRequest`。根 Projectile 的出生位置、Explosion 中心和安全 Teleport 请求目标会在 commit 前冻结，并把同步 slice 记入对应实际 chunk。Explosion 的实体查询包络会展开为 `SynchronousNodeBindings` 中的完整 chunk 集合，每个覆盖 chunk 都预留完整的有界 `ENTITY_SCANS` 上限，并同步累加到 owner、dimension 与 global 总账；这避免边界上的密集实体把查询成本错误归给中心 chunk。G05 尚未实现跨 chunk 地形工作的精确分区，因此可能跨 chunk 的 Explosion terrain 候选或代表性 BREAK 视线会在 reservation/commit 前整体拒绝；未知动态目标也保守 fail closed。中心 `SpellBudgetManager` 在 owner、chunk、dimension、global、tick 和 time-window 维度作全有或全无预留；该账本一律使用单调的服务器 tick，法杖冷却/NBT 仍独立使用世界时间。

`MinecraftEffectExecutor` 只接受事务生成的 execution UUID 和已提交 reservation，不能在提交后自行生成新 UUID 或以未跟踪预算执行世界节点。类型化节点 registry 负责单节点异常隔离和幂等的未使用额度释放；chunk-local slice 由事务提供精确映射，不能猜测其他节点的 chunk 分摊。根 reservation 关闭时保留已承诺的 tick/window 消耗，但释放未被转交给跨 tick 工作的 in-flight 容量。

根投射物在同一事务调用中使用其已提交 reservation 的独立 `AUTHORITATIVE_ENTITIES` local slice，因此不会对根生成重复计入 tick/window。延迟的 Spark/Bomb 实体与 trigger 载荷不能重用已经关闭的根 handle，故在实际释放位置重新申请中央运行时额度。`ServerWorldMutationBudget` 以 execution identity、node path、世界时间、操作种类和序号派生新的 UUID；点操作按实际 chunk 申请，而有界跨 chunk 查询以一个原子 `BudgetRequest` 向每个覆盖 chunk 预留完整结果/检查上限，再执行 `reserve -> commit -> close`。Bomb 的方块 ray 也先验证完整 loaded-chunk 包络和 `BLOCK_CHECKS` DDA 上限，拒绝时不会继续触发实体碰撞。这样延迟操作仍会独立抢占当时的共享额度，可能在服务器压力下被策略拒绝，但不会绕过中央账本或回滚已提交法杖。

## 持久任务交接

`PersistentJobEffectNode` 在根 cast 已提交后才构造 `SpellJobPersistentState`。该记录冻结 execution UUID、owner UUID、dimension、目标 chunk、catalog epoch/hash、节点路径和类型、游标、剩余硬预算、状态/原因及创建与过期 tick；它不保存 live `World`、`ItemStack` 或 handler 引用。

`PersistentJobEffectNode` 将 `recoveryIdempotent` 显式冻结进 `FrozenSpellJobNode`，并在 reservation 前验证完整 step 硬预算。`SpellJobManager` 将根 reservation 中已承诺的 `PERSISTENT_JOBS` slice 转交为 job 生命周期租约，因此根 cast 关闭不会提前释放该容量，也不会对同一次 cast 重复收费。每个实际 handler step 重新申请中央预算；owner 离线、死亡、跨维或目标 chunk 未加载时 job 暂停且不会加载 chunk。达到过期时间或硬预算时取消，handler 消失或抛错时转为 inert。没有同步持久化 pre-effect checkpoint 时，节点与 handler 任一方未声明幂等的 job 会在 admission 前拒绝，避免把普通副作用重放为“恢复”。

服务器启动从 overworld `PersistentState` 恢复记录并重新取得 live job 的生命周期租约。恢复到 `RUNNING` 的记录只有在冻结节点和已注册 handler 都声明幂等时才降为 `PAUSED` 重试；其他 interrupted step 变为 inert。该规则避免把普通世界副作用宣传为 crash-safe exactly-once。当前没有通用的生产 job handler：未显式注册的 `jobType` 会被拒绝/标记 inert，而不会作为未实现后台工作继续运行。

## 测试与扩展

- 纯 JUnit 覆盖 CastBinding 完整匹配、结果状态机、canonical NBT fingerprint、`ResolvedCast -> BudgetRequest` 映射、reservation 额度切分，以及持久 job 的恢复、租约、owner/chunk gate、硬预算和幂等恢复策略。
- 33 项真实 Fabric GameTest 覆盖预算拒绝、打开编辑器、旧 revision 和旧 catalog epoch，以及 reservation 后换栈时的 NBT 字节级不变；另以真实碰撞验证同队 `HEAL` 与 hostile 拒绝。
- `WandCastRequest` v2 与 `WandCastHudSnapshot` 的 JUnit 往返、截断和 canonical hash 验证覆盖网络编码边界。
- 新增 C2S 字段时，先扩展 `ClientCastBinding` 与拒绝测试，再更新 `WandCastRequest` 和客户端 HUD 同步。
- 新增长生命周期节点时，必须注册精确 job handler、保存 execution UUID、catalog epoch/hash 和冻结 mechanics，并实现 reservation 的幂等 close/release、恢复和 GameTest 路径。
