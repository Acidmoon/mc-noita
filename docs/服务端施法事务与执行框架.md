# 服务端施法事务与执行框架

## 责任与边界

`com.mcnoita.wand.server.CastTransaction` 是服务端唯一允许改变法杖状态的施法协调器。纯 evaluator 继续只产生冻结的 `ResolvedCast`；事务层负责把玩家持有的 `ItemStack`、catalog snapshot、预算 reservation 和执行器连接起来。它不重新解释卡牌顺序，不让客户端提交伤害、随机种子、payload 或世界操作。

正式状态机为：

```text
CastIntent
  -> Validate source stack and player
  -> Evaluate on an ItemStack copy against one CatalogSnapshot
  -> Reserve one BudgetRequest with a pre-generated execution UUID
  -> Revalidate CastBinding
  -> Write a replacement ItemStack copy
  -> Revalidate, commit the reservation, and replace the held stack on the server thread
  -> Execute frozen EffectPlan
```

`CastBinding` 绑定 player UUID、hand、slot、item ID/count、StateRevision、spells hash、规范化 NBT SHA-256、catalog epoch/hash 和 sequence。重读中的任一字段不一致都会拒绝本次 cast。`PreparedCast` 只持有隔离副本；`CommittedCast` 才表示原手持栈已被替换。

## 原子性与失败

解码、迁移、冷却检查、外部热栏候选读取和 evaluator 都只接触 `ItemStack.copy()`。因此验证、求值、预算预留或二次 binding 校验失败时，原法杖的 mana、uses、Deck、cooldown、revision 和 NBT 不变。GUI 保存法术会递增 `StateRevision`；打开任意非玩家 ScreenHandler 时直接拒绝施法，避免编辑与 cast 竞争。

预算 reservation 必须在写 replacement 前成功，execution UUID 也必须在 reservation 前生成。提交后，执行器只消费已冻结的 plan；单节点失败被隔离并记录诊断，不回滚已经提交的法杖。当前根投射物的跨生命周期容量仍由 G03 Trigger runtime budget 保护；G05 持久任务会把中央 reservation 的长期释放接到 job/entity 生命周期，而不是把普通副作用称为 crash-safe exactly-once。

## 适配器与网络

`MinecraftWandAdapter.readReadOnly`、`canCastReadOnly` 和 `canonicalNbtStateHash` 是只读边界，HUD 和二次 binding 不得因为 legacy migration 修改真实栈。`MinecraftExternalSpellPoolAdapter` 同样只读取其他热栏法杖的副本。

`CastIntent` 容纳 hand、slot、sequence 和可选 `ClientCastBinding`。协议 v2 将客户端观察到的 revision、catalog epoch/hash 和 NBT state hash 映射到该 binding；`CastTransaction` 在求值前以及两次 commit 前重读时都重新比较它。`MinecraftWandAdapter.stateRevisionReadOnly` 同样不会迁移真实栈。客户端 HUD 同步只作为下一次请求的 optimistic binding，不能授权任何世界效果。

## 预算与执行

`PlanBudgetRequestFactory` 从 `ResolvedCast` 的 action/projectile/payload usage 与 typed effect nodes 生成 `BudgetRequest`。它在施法原点 chunk 记录根投射物、静态实体树、Trigger release、视觉、NBT、网络和代表性世界操作成本。中心 `SpellBudgetManager` 在 owner、chunk、dimension、global、tick 和 time-window 维度作全有或全无预留。

`MinecraftEffectExecutor` 接收事务生成的 execution UUID 和 reservation。它不能在提交后自行生成新 UUID。类型化节点 registry 负责单节点异常隔离和幂等的未使用额度释放；chunk-local slice 由事务提供精确映射，不能猜测其他节点的 chunk 分摊。

## 测试与扩展

- 纯 JUnit 覆盖 CastBinding 完整匹配、结果状态机、canonical NBT fingerprint 和 `ResolvedCast -> BudgetRequest` 映射。
- Fabric GameTest 覆盖预算拒绝、打开编辑器、旧 revision 和旧 catalog epoch 时的 NBT 字节级不变。
- `WandCastRequest` v2 与 `WandCastHudSnapshot` 的 JUnit 往返、截断和 canonical hash 验证覆盖网络编码边界。
- 新增 C2S 字段时，先扩展 `ClientCastBinding` 与拒绝测试，再更新 `WandCastRequest` 和客户端 HUD 同步。
- 新增长生命周期节点时，必须保存 execution UUID、catalog epoch/hash 和冻结 mechanics，并实现 reservation 的幂等 close/release 路径。
