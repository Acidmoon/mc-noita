# 类型化效果执行器

## 边界

`EffectPlan` 现在以不可变 `EffectNode` 列表表达已求值的服务器效果。节点位于纯 `com.mcnoita.spell.plan` 包，不引用 World、Entity、NBT、Registry 或客户端类。兼容期内，`projectiles()`、`sounds()` 和 `recoils()` 仍从类型化节点投影，原有三列表构造器继续可用。

当前节点类型包括 Projectile、Sound、Recoil、Explosion、Field、Summon、Teleport、BlockMutation 与 PersistentJob。Projectile、Sound、Recoil 已有真实 Fabric 执行器。Explosion 使用 `WorldMutationService` 的半径上限、owner、已加载 chunk、world border、保护与可选地形候选路径；这是 Minecraft 的近似爆炸，当前节点的标量 damage 仍等待完整 `DamageProfile` 范围结算迁移。Teleport 只会把已绑定 owner 传送到由服务器朝向推导、已加载且碰撞安全的位置。

BlockMutation 目前只实现单方块、视线内的 `BREAK`：`PLACE`/`REPLACE` 缺少冻结 replacement state，区域选择也没有被节点表达，因此会在提交后安全拒绝并记录限频诊断。Summon 使用带专用原因的 deferred/rejected executor，而不是直接 `spawnEntity`：在持久 job 子系统提供实体 allowlist、生命周期到期和 reload 恢复前，泛型 entity ID 不会生成无期限实体。Field 仍由 deferred executor 接收；生产 `MinecraftEffectExecutor` 已接入 `SpellJobServerService` 的持久化/恢复/预算 sink，但当前没有内置 `SpellJobHandler`，所以未注册的 `jobType` 仍会拒绝并变为 inert。无参 registry 工厂保留拒绝型 sink 仅用于 legacy/test。以上状态均不表示对应法术族已经完成迁移。

## 执行流

```text
Committed ResolvedCast + transaction execution UUID + optional BudgetReservation
  -> MinecraftEffectExecutor
  -> EffectExecutionContext
  -> EffectExecutorRegistry
  -> typed EffectNodeExecutor
```

`MinecraftEffectExecutor.execute(player, resolvedCast, executionId, reservation)` 是正式入口。事务必须在预算预留前生成 UUID，并使其与 reservation 的 UUID 一致。旧两参 overload 仅保留给 legacy/test 路径；它不能成为正式服务器施法事务的调用点。

每个 registry dispatch 在本节点范围内捕获 `RuntimeException`。失败不会回滚已提交的法杖状态，也不会重放其他节点；诊断按节点类型和阶段限频。根 Projectile executor 在开始实体分配前保留现有 Trigger runtime-tree 下界分配，若该旧保护拒绝一个根节点，其余 Sound/Recoil 节点仍可独立执行。

当 transaction 提供 `BudgetReservation` 时，registry 会在节点异常、缺失 executor 或 deferred executor 返回后，以 `effect-node/<stage>/<nodePath>` 作为幂等 key 请求释放未使用额度。默认 mapper 仅释放可证明没有 chunk-local 分摊的受限 slice；WP2 transaction 可通过五参 overload 传入精确的 `EffectNodeBudgetReleaser`，为多节点或 chunk-local 计划提供精确切片。没有 reservation 的 legacy/test 执行严格不释放额度。

即时 Explosion、Teleport 与 BlockMutation context 各自持有由该 reservation 根 `BudgetRequest` 预分配的 `WorldMutationBudgetCounter` 切片。切片互不重叠，因此一个失败节点释放其中心未用额度后，后续节点不能重用它的本地容量；它们共同把方块检查/修改、实体扫描和实体生成限制在已接受计划的总量内。该执行期拒绝阀不会重新预留、释放或改变 `SpellBudgetManager` 的中心账本。

`PlanBudgetRequestFactory` 在 commit 前冻结根 Projectile 出生点以及安全 Teleport 请求目标的 chunk，并将对应 local slice 预留到该实际 chunk。Explosion 中心也使用冻结出生点；但半径候选、视线 BREAK 的最终命中点和跨 chunk 实体查询尚未有精确的多 chunk 分摊模型。它们一旦跨出自身已预留的 chunk 就 fail closed，不能被误记到施法原点或绕过目标 chunk 配额。

## Trigger 复用

根 Projectile 节点先由 `ProjectilePlanPayloadAdapter` 变成冻结的 `NoitaProjectilePayload`，随后交给 `ProjectileDispatcher`。`TriggerPayloadSpawner` 使用同一 dispatcher，因此 Trigger 释放和根投射物共享同一个实体生成入口与现有世界修改策略边界。

## 后续扩展

后续 Explosion 的范围伤害、Field、Summon 生命周期、区域 BlockMutation 与 PersistentJob 仍必须使用 central budget reservation、WorldMutationPolicy、所有者/维度/区块校验和持久 job 语义。不得通过替换受限 executor 或 deferred executor 为直接 `World` 调用绕过这些边界。
