# 世界修改与伤害策略（G05-6/7）

## 责任边界

`com.mcnoita.world.mutation` 是服务端法术进行方块修改、危险实体生成、爆炸地形候选和大范围实体查询的唯一边界。它不解释卡牌、不扣除法力、不重算 Trigger 载荷。

`com.mcnoita.spell.damage` 是直接伤害与治疗的唯一服务端边界：`SpellDamageService` 将冻结的 `DamageProfile` 转换为 Minecraft `DamageSource`，`HealingService` 执行正值治疗。实体类不得直接构造 `indirectMagic`、调用旧 `NoitaSpellDamage` 或直接调用 `LivingEntity.heal`。

当前入口是：

```text
SparkBoltProjectileEntity / BombEntity
  -> WorldMutationContext
  -> WorldQueryService 或 WorldMutationService
  -> WorldMutationPolicy
  -> Minecraft ServerWorld

ProjectilePlan(DamageProfile)
  -> NoitaProjectilePayload(DamageProfile, NBT v5; v4 compatibility)
  -> SparkBoltProjectileEntity
  -> SpellDamageService / HealingService
  -> Minecraft ServerWorld
```

`WorldMutationContext` 绑定 source、owner UUID、预期维度、冻结 execution identity 和预算钩子。实体与载荷 context 必须持有 bound identity；缺失预算或未绑定 identity 会 fail closed，不能回落到未跟踪额度。执行时重新从服务器 PlayerManager 解析 owner，并要求 owner 在线、未移除、存活且仍在同一维度；因此 owner 登出、死亡、跨维度或缺失 owner 都不能继续进行破坏性操作。

## 默认保守策略

破坏性方块操作、爆炸和危险实体生成必须同时满足：

- 逻辑服务端、预期维度、build limit、world border 和目标 chunk 已加载。
- owner 是当前在线且仍在同一 `ServerWorld` 的玩家。
- `mobGriefing` 为 true。G05 明确把它应用于玩家施法、召唤、反射投射物和离线 owner 的遗留投射物。
- spawn protection 与 `ServerWorld.canPlayerModifyAt` 均允许。
- 所有已注册 `ProtectionAdapter` 都允许。
- 方块不在 `mc-noita:spell_unbreakable`，硬度有限、非负且不高于 20。
- 预算钩子接受本次检查、修改、生成或查询。

没有 claims adapter 时不会声称通用领地兼容；默认仍由原版 spawn protection 和上述保守拒绝路径保护。

临时光是唯一的短生命周期视觉例外：`NoitaTemporaryLightManager` 只可通过
`WorldMutationService.placeTemporaryLight/clearTemporaryLight` 写入空气或已有 `minecraft:light`，绝不覆盖普通方块。它不要求在线 owner 或 `mobGriefing`，以便投射物轨迹在 owner 登出后仍能清理；但仍检查 loaded chunk、build limit、world border、保护适配器和预算钩子。目标区块未加载时保留待清理记录而不强加载。
待清理位置以 v2 `TemporaryLightPersistentStateStore` 保存维度、坐标、expiry 和 light level fingerprint；服务器重启后仅在目标 world/chunk 已加载时再次走相同策略清理，绝不为视觉残留加载区块。若同位置的 light level 已被后来写入者改变，manager 只移除自身账本项而不删除该方块。v1 记录仍可读取，但因没有 fingerprint 只保留一次旧兼容清理行为；future ledger 变为只读并拒绝创建新的临时光，避免产生无账本残留。

`data/mc-noita/tags/blocks/spell_unbreakable.json` 保护基岩、屏障、命令/结构方块、末地传送方块和强化深板岩。服务器数据包可扩展该 tag。

## 有界操作

`WorldQueryService` 要求整个 query Box 的 chunk 已加载，最大直径为 64、最大体积为 262,144、最多跨 25 个 chunk，结果最多 128 个实体；几何上限会在任何 chunk 枚举之前检查，并通过限额 `collectEntitiesByType` 在 EntityLookup 层停止遍历。它的 `tryEntities` 能区分“合法空结果”和预算/策略拒绝，因此被拒绝的 Explosion 不会伪装成成功的零伤害节点。延迟实体的 `ServerWorldMutationBudget` 对跨 chunk Box 以单一原子请求向每个覆盖 chunk 保守预留完整结果上限，而不是只归属 source chunk；任一 chunk 超限时不会留下部分额度。根 `ExplosionEffectNode` 在 commit 前冻结同一包络并逐 chunk 预留其实体扫描 slice，执行时仅允许该节点访问已冻结集合。G05 尚未有跨 chunk terrain partition：可能触及多个 chunk 的 Explosion 地形候选或代表性 BREAK 视线会在 transaction 预检中拒绝，不能等到 WandState commit 后由 local slice 静默丢弃。`WorldMutationService.explode` 限制半径为 16，不调用没有结果上限的 vanilla explosion；它以冻结的 `DamageProfile`、距离衰减、归属和友伤策略执行有上限实体阶段，再最多枚举 512 个球形方块候选，并逐个经过策略和预算后破坏。原版 explosion fire 被延后，法术放火必须逐块走 `replaceBlock`。

方块 ray 同样只能走 `WorldQueryService.raycast`：端点、build limit、world border 和完整 chunk 包络必须先通过，长度最多 64 方块，DDA 的保守上限最多 193 个 `BLOCK_CHECKS`。只有预算已原子取得后才调用 Minecraft 原始 raycast；被拒绝的 Bomb 本 tick 不再继续使用未验证路径释放 Hit 载荷。

这属于 Minecraft 等效实现，不承诺逐像素 Noita 爆炸地形。实体与载荷运行时使用 `ServerWorldMutationBudget` 向共享 `SpellBudgetManager` 为每项实际操作重新预留：`BLOCK_CHECK -> BLOCK_CHECKS`、`BLOCK_MUTATION -> BLOCK_MUTATIONS`、`ENTITY_QUERY` 与爆炸的查询阶段 -> `ENTITY_SCANS`、`ENTITY_SPAWN -> AUTHORITATIVE_ENTITIES`。请求带 owner、当前维度与 source/载荷当前位置的 chunk，使用派生 operation UUID 后执行 `reserve -> commit -> close`；因此已提交操作释放 in-flight 容量但保留 tick/window 历史。根投射物同步生成消费其已提交 local slice；只有根关闭后的延迟操作重新预留，不能据此宣称精确的跨生命周期 per-cast 分摊。

## 现有保护与限制

Trigger v3 的 frozen payload、runtime budget、execution identity 和正常 save/reload at-most-once 语义保持不变。该策略不把普通副作用升级为 crash-safe exactly-once，也不创建跨 tick job；这些属于 G05-8。

## 多通道伤害与治疗

`DamageProfile` 是纯 Java、不可变的通道金额表。它不引用 Minecraft、NBT、世界、实体或注册表；相同冻结计划总是产生相同 profile。历史标量 `damage` 仍保留为兼容投影，但只表示 `PROJECTILE` 通道。旧构造器、旧 catalog 与 v3 payload 均按这个规则迁移，绝不从物品名猜测元素类型。

| 通道 | 当前 Minecraft 映射 | 适配等级 | 说明 |
|---|---|---|---|
| `PROJECTILE` | `mc-noita:projectile` | equivalent | 所有历史标量伤害的默认通道。 |
| `EXPLOSION` | `mc-noita:explosion` | approximate | 直接伤害与受策略约束的地形爆炸分开执行。 |
| `FIRE` | `mc-noita:fire` | approximate | 直接火焰伤害不替代 `setOnFireFor`。 |
| `ELECTRICITY` | `mc-noita:electricity` | approximate | 导电、雷击和状态效果仍由有界执行器单独处理。 |
| `DRILL`、`SLICE` | 各自 damage type | approximate | 只表示伤害归类，不授权方块破坏。 |
| `ICE` | `mc-noita:ice` | approximate | 冻结 ticks 与减速状态单独结算。 |
| `TOXIC`、`POISON` | 各自 damage type | approximate | 中毒状态不会被折叠成直接伤害。 |
| `CURSE`、`HOLY` | 各自 damage type | approximate | 目前用于显式代表场景；完整法术分类属于后续 catalog 工作。 |

`SparkBoltProjectileEntity` 保存 payload 的完整 profile。主命中会将暴击倍率施加到全部直接伤害通道；随机伤害修正仅调整 `PROJECTILE`，以保持旧标量行为。Darkflame 的周期伤害显式使用 `FIRE`，Cursed Orb 的周期伤害显式使用 `CURSE`；其余遗留二次伤害暂时走 `PROJECTILE` 并保持近似标记。火焰、冰冻、中毒、凋零、减速等状态效果仍是独立结算步骤。owner 缺失、已移除、死亡、跨维或旁观时，Spark 的有害伤害、状态和击退路径全部 fail closed，不能把 null owner 当作无条件友伤许可。

`SpellDamageService` 使用投射物作为 direct entity、当前 `getOwner()` 作为 owner attribution。普通投射物默认拒绝自伤和队友伤害；`friendlyFire` 同时允许两者。范围调用可显式允许 owner 而继续保护队友，避免把旧 `includeOwner` 语义错误地扩展为全队友伤害。相同 target gate 也包住有害的命中后状态、击退、火焰与持续场，不能通过“状态与直接伤害分开结算”绕过队友保护。穿透与多次碰撞仍经过同一服务，服务会在成功命中时清除目标的伤害无敌帧，保持既有 Noita 风格的重复命中行为。反射后的归属以实体当前 owner 为准，客户端不能提交 owner、damage profile 或伤害金额。

治疗不使用负伤害。`HealingService` 只在逻辑服务端接受有限正值，默认只治疗 owner、本体或队友；治疗敌人必须由调用点显式选择。Heal projectile 和血雾当前以 `friendlyFire` 作为“允许非盟友治疗”的显式开关，Friend Fly 只治疗 owner。这个保守目标筛选与早期“范围内所有生物都治疗”的实现不同，属于有意的 Minecraft 安全适配。

## DamageProfile v4 兼容性与 v5 写入

冻结 `NoitaProjectilePayload` 当前以 v5 写入 `DamageProfile` compound，键为 `DamageChannel` 名称，金额必须是有限、非负、每通道不超过 `MAX_ABSOLUTE_PROJECTILE_DAMAGE` 的数字；未知通道、非数字、NaN、无穷或超限值会让整棵 payload 变为 inert。v4 引入的旧 `Damage` 标量仍写成由 profile 推导的兼容/诊断镜像，但 profile 是机械权威，v5 decoder 会拒绝缺失或不匹配的镜像。

`NoitaNbtSchema` 的 v3 -> v4 migration 仅处理 `PROJECTILE_PAYLOAD`，将旧标量转换为 `{PROJECTILE: max(0, Damage)}`；v4 -> v5 保留该冻结机制并推进统一 schema marker。实体根 NBT 不伪造 profile，内嵌 `FrozenPayload` 由自己的版本链迁移。v3 已引入完整 frozen-tree 契约；当前 v5 根还拒绝嵌入旧版本子树，避免新旧树混合。

## 测试与扩展

- `WorldMutationBudgetCounterTest` 验证预接受的纯预算钩子不会在失败时部分扣除；`ServerWorldMutationBudgetTest` 覆盖实体/载荷的中央操作映射、跨 chunk 原子预留、派生 UUID、tick/window 累积、in-flight 释放、未绑定 identity 拒绝和 `/time set` 后仍使用单调服务器 tick。
- `WorldMutationArchitectureTest` 与 `checkMainServerArchitecture` 阻止 Spark/Bomb 直接调用危险世界 API、原始 `World.raycast` 或绕开 `WorldQueryService` 的方块读取。
- `WorldQueryServiceTest` 覆盖实体收集上限、ray DDA 边界与无界 vanilla 查询/爆炸路径禁用；`ImmediateWorldBudgetAllocatorTest` 和 `PlanBudgetRequestFactoryTest` 覆盖跨 chunk Explosion 的预留、允许包络与拒绝第三个未预留 chunk；`NoitaTemporaryLightManagerTest` 覆盖 v1 兼容、v2 light-level cleanup ledger、上限、future-format fail-closed 和重试账本。
- `DamageProfileTest` 覆盖 profile 的纯组合、缩放、替换和非法值拒绝；`FrozenTriggerPayloadNbtTest` 覆盖旧标量、双通道往返、v3 -> v4 迁移以及损坏 profile 拒绝。
- `checkMainServerArchitecture` 同时拒绝实体类中的旧 `NoitaSpellDamage.apply`、`indirectMagic`、直接治疗和常见原始 `Entity.damage` 调用。
- 未来 adapter 应只实现 `ProtectionAdapter.allows`，不得从 adapter 强加载 chunk 或修改世界。
- 新增世界副作用必须先选择 `WorldQueryService` 的有限查询，再调用 `WorldMutationService`，不能回到实体内的 `World` 直接调用。
- G05 将第三方实体的 Homebringer/Teleportation Field relocation 和 legacy mob/item summon 分支显式 deferred；它们不能在没有冻结目标、owner tag、数量和生命周期 policy 的情况下回退到原始世界调用。
