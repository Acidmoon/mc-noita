# Network Protocol

## Protocol Version

`NoitaNetworkProtocol.VERSION` is currently `2`. Every C2S request and both
S2C HUD streams start with this version. Unknown versions are ignored before
world work is scheduled.

## C2S Requests

`WandCastRequest` contains only client-observed binding values:

1. Protocol version.
2. Monotonically increasing sequence number.
3. Hand.
4. Inventory slot.
5. Client-observed wand state hash.
6. Client-observed `StateRevision`.
7. Client-observed catalog epoch.
8. Canonical SHA-256 catalog hash.

The NBT component of the state hash is a recursively key-sorted SHA-256
fingerprint, not `NbtCompound.hashCode()`. These fields detect stale requests
only; they are not authority. The server rereads the selected main-hand stack,
verifies the slot, item type, revision, state hash, epoch/hash, cooldown and
eligibility, then the transaction repeats the binding check before and after
reservation. Mana, cards, uses, random seed, payload and all world effects are
always derived on the server.

`HoverInputRequest` contains protocol version, a separate sequence number,
jump state, and sideways/forward inputs. Decoder input must be finite; finite
movement is clamped to `[-1, 1]` before it reaches hover motion.

Both decoders reject oversized packets, truncated fields, trailing bytes,
negative sequence/revision/epoch values, invalid hand or slot values, and
non-canonical catalog hashes. Server actions are scheduled on the logical
server thread after decoding.

## Replay And Rate Limits

`NoitaRequestGuard` has independent per-player token buckets:

- Casting: capacity 8, refill 8 requests per second.
- Hover input: capacity 12, refill 12 requests per second.

Each stream rejects duplicate and decreasing sequence numbers. State is removed
on player disconnect. A rate-limited request consumes its new sequence number,
which prevents an attacker from replaying it when the bucket later refills.

The client sends hover input only when state changes or after a 20-tick
heartbeat. It does not unconditionally emit a packet every client tick.

## S2C Validation

Wand HUD packets carry version, timing mode/progress, mana state, current wand
revision, catalog epoch/hash and state hash. The client stores that server
projection solely to echo it in its next cast intent; it does not derive or
choose mechanics from it. Hover sync packets carry version and energy values.
Clients reject wrong versions and trailing data, clamp progress to its declared
total, clamp mana and energy to their maxima, and reject non-finite mana.
