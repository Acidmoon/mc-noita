# Network Protocol

## Protocol Version

`NoitaNetworkProtocol.VERSION` is currently `1`. Every C2S request and both
S2C HUD streams start with this version. Unknown versions are ignored before
world work is scheduled.

## C2S Requests

`WandCastRequest` contains only:

1. Protocol version.
2. Monotonically increasing sequence number.
3. Hand.
4. Inventory slot.
5. Client-observed wand state hash.

The hash detects stale requests only. It is not authority: the server rereads
the selected main-hand stack, verifies the slot and item type, checks the hash,
checks cooldown, and evaluates mana, cards, uses, and projectile values from
its current item state.

`HoverInputRequest` contains protocol version, a separate sequence number,
jump state, and sideways/forward inputs. Decoder input must be finite; finite
movement is clamped to `[-1, 1]` before it reaches hover motion.

Both decoders reject oversized packets, truncated fields, trailing bytes,
negative sequences, and invalid hand values. Server actions are scheduled on
the logical server thread after decoding.

## Replay And Rate Limits

`NoitaRequestGuard` has independent per-player token buckets:

- Casting: capacity 8, refill 8 requests per second.
- Hover input: capacity 12, refill 12 requests per second.

Each stream rejects duplicate and decreasing sequence numbers. State is removed
on player disconnect. A rate-limited request consumes its new sequence number,
which prevents an attacker from replaying it when the bucket later refills.

The client sends hover input only when state changes or after a 20-tick
heartbeat. It does not unconditionally emit a packet every client tick.

## S2C Validation

Wand HUD packets carry version, timing mode/progress, and mana state. Hover
sync packets carry version and energy values. Clients reject wrong versions and
trailing data, clamp progress to its declared total, clamp mana and energy to
their maxima, and reject non-finite mana.
