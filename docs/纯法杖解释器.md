# 纯法杖解释器（G01-G03）

## 责任与边界

G01 将一次施法拆成两个严格分离的阶段：

```text
ItemStack/NBT + 当前服务器时间
       | MinecraftWandAdapter
       v
WandDefinition + WandState + SpellCatalog + elapsed + seed
       | WandCastEvaluator（纯 Java）
       v
ResolvedCast（下一状态 + 冻结 EffectPlan）
       | 提交 WandState
       v
MinecraftEffectExecutor（实体、后坐力、声音）
```

`com.mcnoita.wand.model`、`wand.eval`、`spell.action` 与 `spell.plan` 不导入 Minecraft、Fabric、实体、物品、NBT、网络、世界或客户端类。`PureEvaluatorArchitectureTest` 会扫描这四个包，防止边界倒退。

`NoitaWandCaster` 只保留服务器编排职责：验证主手法杖和冷却、读取适配模型、执行纯求值、提交成功的下一状态，再执行冻结的计划。若执行器节点失败，已经提交的法力、牌堆和次数不回滚；执行器会隔离该节点并按节点类型限频记录诊断。

## 主要类型

- `WandDefinition`：不随 cast 改变的洗牌、Spells/Cast、时序、法力、容量、散射、速度与 Always Cast 定义。
- `SpellCardState` 与 `CardRef`：每张配置卡的稳定槽位身份、法术 ID、剩余次数。重复法术仍是独立卡片。
- `DeckState`：不可变 Deck、Hand、Discard 及全部卡片映射；构造时强制每张配置卡正好位于一个牌堆。
- `WandState`：不可变牌堆、法力、剩余 Cast Delay/Recharge、重载标记、revision 与状态哈希。
- `ShotState`：不可变当前 Shot 修正累计器。`applyModifier` 返回新值，因此多重施放共享同一 Shot，而 Trigger 载荷获得独立 Shot。
- `SpellDefinition` 和显式 `SpellAction`：目录使用 `DrawAction`、`ModifyShotAction`、`AddProjectileAction`、`BeginTriggerAction`、`CallSpellAction`、`DuplicateHandAction`、`RefreshWandAction`、`RandomSpellAction` 与支持 `ADD`/`SET` 的 `TimingAction`，不再通过物品类别猜测动作。`UseConsumptionPolicy` 明确 Hand 弃牌时的自动次数规则。
- `DrawRequest`、`DrawOutcome` 与 `DrawOrigin`：Initial、Action、Payload、Permanent Draw 不再共享布尔参数；输出保留成功卡、失败原因、Wrap、Deck 耗尽和 Reload 请求。
- `ResolvedCast`：唯一成功输出，含下一份 `WandState`、`EffectPlan`、次数结果、最终时序、seed、catalog epoch/hash、预算用量与结构化诊断。

`EffectPlan` 的 `ProjectilePlan` 已冻结伤害、速度、寿命、散射偏移、修正器和 Trigger 子树。G03 将原先分散的 Trigger mode/delay/payload 字段收束为 `TriggerPlan` 与 `PayloadPlan`，每个节点具有稳定 `nodePath`。实体生成后不会按法术 ID 重新读取当前目录。

## 时间与随机数

纯解释器统一使用 `NoitaDuration`（60 Hz Noita 帧）。`MinecraftTimeAdapter` 是唯一 Minecraft tick 转换位置：读取旧 NBT 时把 tick 转为帧，写回冷却和实体载荷时向上取整为 tick，并应用调用点所需的最小 tick。

旧 `NoitaProjectilePayload` 的寿命和 Trigger 延迟历史上已经是 Minecraft tick；`LegacySpellCatalogAdapter` 在边界把它们显式映射为等价的 Noita 帧，以保证旧存档和现有实体路径兼容。后续目录迁移应直接记录规范化 Noita 帧，而不是延续这个兼容字段。

`CastRng` 由调用者提供根 seed，并为 `shuffle`、`random-spell`、`spread` 使用独立命名子流。新增其他随机用途不会改变同一输入的 Deck 顺序。投射物散射偏移已在 `ProjectilePlan` 中结算，执行器不再为初始散射调用世界随机数。

## 预算与失败行为

默认单次纯计划预算为：2048 个 action step、128 个逻辑投射物、128 个 Trigger 载荷节点、16 层 Trigger payload depth、32 个第一次完整树实体、32 个第一次完整树 release event、2 层 recursive Call。每个 `PayloadPlan` 还最多容纳 32 个逻辑投射物，和持久化子项上限保持一致。节点数与深度是独立上限；Draw、action、Call、投射物和载荷分支均通过 `WandCastSession` 计数。冻结完成后，解释器还会计算物理树的首次释放下界并在 WandState 提交前拒绝超过实体/事件预算的树，避免预付 payload 在命中时才因容量不足丢失。

达到预算时，解释器返回：

- `ResolvedCast.Status.REJECTED`。
- 原始 `WandState` 与空 `EffectPlan`。
- 含错误码、当前法术、动作路径、实际用量和上限的 `CastDiagnostic`。

因此不会提交半损坏 Deck、部分扣费或未受限的递归树。跨玩家、区块、维度和全局 tick 的预算预留仍属于后续服务器预算 Goal；G01 仅建立纯单次计划上限。

## 当前 Legacy 兼容层

`LegacySpellCatalogAdapter` 从已注册 `NoitaSpellItem`/`NoitaProjectileSpellItem` 建立不可变目录。它保留已有 registry ID、`NoitaWandCastState`、`Deck`、`Discarded`、`CurrentMana`、冷却与 `RemainingUses` 键。

`LegacyWandStateAdapter` 将旧的槽位整数牌堆形状和纯 `WandState` 相互转换；`MinecraftWandAdapter` 负责其 NBT 与 `ItemStack` I/O。Cast State v2 将 G01 的“Recharge 到期才重建 Deck”迁移为 G02 的“施法提交时已重建 Deck”，并以稳定迁移 seed 处理旧 shuffle 状态。该转换的无世界回归测试覆盖槽位、Deck/Discard、法力、每卡剩余次数和 G01 schema 迁移。

当前适配器已覆盖现有 Spark Bolt、Bomb、修正器、Multicast、Trigger、Alpha、Gamma、Duplicate、Wand Refresh 与随机投射物的可执行桥接。它是 G01 的兼容基础，不是这些法术都已完成 Wiki 语义的声明。

## 已验证的语义与待续工作

G01-G03 测试验证确定性、洗牌卡片排列、卡片守恒、法力/次数不足、共享 Multicast ShotState、Trigger ShotState 隔离、payload Recharge 全局性、随机候选限制、Call/递归预算、输出不可变性与纯包依赖门禁。G03 额外验证 payload 在 cast 时支付、外层与嵌套 payload 的 nodePath、Timer 元数据、独立 depth/node 预算，以及三层 Trigger 树。`WikiGoldenFixturesTest` 不再禁用，已实际调用解释器验证 Double + Damage Plus、初始 Spells/Cast 不 Wrap，以及 Gamma 不自动支付目标法力或次数。

G02 已验证非乱序与固定 seed 洗牌、Initial Draw 不 Wrap、Action Draw Wrap、末卡失败不立即 Wrap、失败重试、Reload 重排、两次 Wand Refresh、正负法力、自然回蓝、Hand 弃牌次数策略、Always Cast 的法力/次数/Draw 例外、并行 Cast Delay 与 Recharge，以及 `TimingAction.SET`。`runGametest` 还验证真实三张 Spark Bolt、Spells/Cast=2 的连续两轮实体、法力、Deck 和冷却与纯输出一致。完整转换表与证据见 `基础施法语义.md`。

以下仍明确属于后续 Goal：

- 后续 Copy/Call Goal：完整 Greek/Copy/Divide/Add Trigger 的目标选择、递归标记和手动次数规则。
- 后续执行器 Goal：世界级预算预留、材料/地形权限、持久 Trigger one-shot 账本与多维度生命周期。

## 扩展步骤

1. 先在纯 `SpellDefinition` 中选择或新增一个明确 `SpellAction`，并为其写无世界单元测试。
2. 若动作输出世界效果，扩展 `EffectPlan` 节点并冻结所有机械参数；禁止执行时按 ID 再解释目录。
3. 在 `MinecraftEffectExecutor` 添加受限的世界映射与逐节点失败隔离。
4. 仅在 `LegacySpellCatalogAdapter` 或未来数据目录层接触 Registry/ItemStack；保持纯包门禁通过。
5. 对新的 Noita 可观察语义增加 Wiki golden fixture，并记录任何 Minecraft `equivalent` 或 `approximate` 差异。
