# 多层预算与持久任务

## 范围与边界

G05 将一次已提交施法的长期工作表示为冻结的
`SpellJobPersistentState`。纯法杖解释器仍只产生 `EffectPlan`；服务端负责
将 `PersistentJobEffectNode` 提交给 `SpellJobServerService`，并在逻辑服务器
线程上调度。持久任务不保存 `World`、实体、物品栈、Registry raw ID 或当前
catalog 的可变对象。

每条记录保存 execution UUID、owner UUID、目标维度和 chunk、catalog epoch/hash、
冻结节点、cursor、剩余硬预算、状态以及创建/过期 tick。catalog hash 只是冻结
证据；恢复时不会按 ID 重新查询或替换为当前 catalog 的法术定义。

`SpellJobPersistentStateStore` 使用主世界的 `PersistentStateManager` 保存记录。
服务启动时恢复记录，结束 tick 时保存规范化快照，停止服务器前保存一次并关闭
运行时 lease。存储格式通过全局 `NoitaNbtSchema` v5 管理。

## 预算生命周期

中央 `SpellBudgetManager` 管理 owner、chunk、dimension 和 global 的 in-flight、
tick、window 额度。普通持久任务使用两类额度：

- `PERSISTENT_JOBS` 是记录生命周期 lease。施法根 reservation 中已预留的
  origin-chunk slice 会在 commit 后转移为一个派生 UUID 的 child lease；转移不
  释放或重新扣除任何 scope，根 reservation 关闭后 child lease 继续占用 in-flight
  容量。没有根 reservation 的恢复或服务端直接提交会重新申请一个新 lease。
- `CROSS_TICK_JOB_STEPS` 和节点声明的逐步成本在每次 handler 调用前重新申请。
  一次已提交的 step reservation 关闭后保留该 tick/window 的消费历史，只释放
  in-flight 容量。

终态 `COMPLETED`、`CANCELLED`、`INERT` 会幂等关闭 lifetime lease。预算拒绝时，
任务保持 `PAUSED`，不会调用 handler、强加载 chunk 或部分执行世界操作。

## 状态机与恢复

状态是 `QUEUED`、`PAUSED`、`RUNNING`、`COMPLETED`、`CANCELLED`、`INERT`。`PersistentJobEffectNode.recoveryIdempotent` 是冻结的 admission 声明，不是 handler 的运行时猜测；其完整 maximum-step 硬预算在施法 reservation 前验证。
每个真正的 handler 调用先把内存状态置为 `RUNNING`，然后在该 step 的 reservation
提交后执行；结果会消费恰好一个 cursor 和其冻结的逐步预算。剩余预算必须严格等于
初始预算减去 `cursor * perStepBudget`，因此损坏 NBT 不能伪造更多工作额度。

恢复 `RUNNING` 记录时，只有以下两个条件同时满足，才会降为 `PAUSED` 并允许再次
尝试：

1. 冻结节点声明 `recoveryIdempotent=true`。
2. 当前注册的 `SpellJobHandler` 也声明 `isRecoveryIdempotent()`。

任一条件不满足，记录变为 `INERT`，不会调用 handler。没有同步 durable pre-effect checkpoint 的当前实现还会在 admission 前拒绝非幂等节点/handler 组合。这个机制不是 crash-safe exactly-once 保证：进程可能在世界副作用完成而状态保存之前崩溃。只有显式幂等的 handler 可以安全重试；非幂等世界副作用必须保守地停止。

## NBT 验证与损坏数据

v5 codec 对 UUID、SHA-256 catalog hash、枚举、NBT 类型、树深度、字节数、参数数、
预算 map 和 cursor/预算关系进行严格校验。future schema 或未知预算类型永远不会被
解码为可执行任务。

可识别 execution/owner/dimension/catalog 绑定但机械字段损坏的记录会被替换成无处理器
的 `INERT` 诊断记录并在后续保存时规范化。无法安全读取身份字段、整份 store 超限或
树结构损坏到无法识别的记录只能记录日志并丢弃；它们从不进入 scheduler。

## 世界门控

`SpellJobServerService` 在每个 tick 为任务提供以下只读门控：

- owner 必须在线、存活、非旁观，且仍在记录的维度。
- 目标 `ServerChunkManager.isChunkLoaded(x, z)` 必须为真。

任一条件不满足时状态为 `PAUSED`。实现不会调用 getChunk、不会请求 ticket，也不会为
任务强加载区块。实际 handler 仍必须通过 `WorldMutationPolicy` 和世界查询服务执行其
自身的权限、边界、方块和实体预算检查。

## 当前启用状态与扩展方式

生产服务、PersistentState、共享预算账本和 effect executor sink 已接通，但目前没有
内置 `SpellJobHandler`。因此未注册的 `PersistentJobEffectNode` 在提交处被诊断并返回
`INERT`，不会保留 live job 或 lifetime lease。未来迁移一个跨 tick 效果时应：

1. 定义稳定 job type 与冻结参数，并给出精确 `perStepBudget`。
2. 注册受限 handler；只有可重放的实现才声明 idempotent。
3. 在 handler 内使用世界修改策略，不直接访问危险世界 API。
4. 为正常完成、owner 断线、chunk unload、预算拒绝、save/reload、损坏 NBT 与
   non-idempotent recovery 添加 JUnit 和 GameTest。

不要把该基础设施描述为对普通世界副作用的 exactly-once 事务，也不要将暂停任务视为
可绕过 owner、chunk、保护或全局预算的后台队列。

当前一个 root cast 最多只能包含一个 `PersistentJobEffectNode`。G05 的 durable key
仍是 root execution UUID，多个节点会在预算规划阶段、法杖状态写入之前被拒绝；后续
若要支持多个长期节点，必须先在 schema、lease 和恢复状态中引入稳定的 per-node job ID。
