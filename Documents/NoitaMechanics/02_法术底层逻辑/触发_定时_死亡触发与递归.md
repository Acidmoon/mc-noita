# 触发、定时、死亡触发与递归

## 三种常见触发器

| Lua 调用 | 玩家语义 | 底层含义 |
|---|---|---|
| `add_projectile_trigger_hit_world(file, n)` | 触发 | 投射物碰撞世界/目标时释放预先写入的 n 张 payload。 |
| `add_projectile_trigger_timer(file, delay, n)` | 定时 | 投射物存在指定帧数后释放 payload。 |
| `add_projectile_trigger_death(file, n)` | 死亡/失效触发 | 投射物死亡、寿命结束或被销毁时释放 payload。 |

这三类函数都会在创建投射物时调用 `draw_shot(create_shot(n), true)`。payload 的抽牌、法力、次数和修正器结算发生在施法时，不是在触发发生时。

## Payload 也是一个 shot

触发器内部创建的是嵌套 shot。它拥有自己的 `c` 状态，能继续吃修正器、多重释放，甚至继续创建触发器。这样就形成 Noita 标志性的树状施法结构：

```text
外层投射物
  └─ trigger
      └─ payload shot
          ├─ 修正器
          ├─ 多重释放
          └─ 内层投射物
```

## 递归限制与特殊例外

`gun.lua` 初始化 `recursion_limit = 2`，说明脚本层对部分递归调用有保护。另一些特殊法术会自己管理递归参数，例如条件法术、希腊字母法术、复制类法术。

实现时不能只用“触发器最多嵌套两层”这种简单规则。更稳妥的做法是给施法解释器一个递归预算，并让每类特殊 action 显式决定是否消耗预算。

## Add Trigger / Add Timer / Add Death Trigger

`ADD_TRIGGER` 不是普通修正器。它会查看 deck 中后续卡牌，跳过部分修正器/被动/特殊/多重释放，并把找到的投射物改造成触发型投射物。如果后续 deck 中还有可作为 payload 的法术，它会创建 trigger；否则会直接执行目标 action。

这类法术需要能操作 deck、discarded、uses_remaining 和 `dont_draw_actions`，所以在 MC-noita 中应放到“抽牌解释器”层实现，而不是放到单个 ProjectileEntity 里。

## 设计建议

Minecraft 侧可以把触发器表示为：

- `ProjectilePayload`：已结算好的嵌套 shot 数据。
- `TriggerMode`：hit、timer、death。
- `TriggerDelayTicks`：timer 用。
- `RecursionDepth`：防止无限触发树。

触发发生时只负责释放 payload，不再重新抽牌或重新花费法力。
